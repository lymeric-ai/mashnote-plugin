이 Claude 대화를 MashNote 활동 피드에 기록하라. 대화가 길고 주제가 여러 개면
주제별로 나눠 여러 건으로 남길 수 있고, 이미 기록한 단위는 다시 만들지 말고 갱신한다.

## 1. 워크스페이스 선택
`list_workspaces`를 호출해 어디에 기록할지 사용자에게 물어봐라. 이 대화에서 이미
골랐으면 그것을 쓰고, 워크스페이스가 하나뿐이면 묻지 말 것.

## 2. 시각(occurredAt) 확보 — Claude Code 한정
대화 컨텍스트에는 메시지별 시각이 없으므로, 실제 시각은 **transcript 파일**에서 읽는다.

- `~/.claude/projects/` 아래에서 **가장 최근에 수정된 `*.jsonl`**이 지금 이 세션의
  transcript다. 그 파일을 읽어라(`type`이 `user`/`assistant`인 엔트리마다 `timestamp`가
  ISO 8601로 들어 있다).
- 각 기록의 `occurredAt`은 그 내용이 **다뤄지기 시작한 첫 메시지의 `timestamp`**로 잡는다.
- transcript를 못 찾으면 `occurredAt`을 **생략**한다(서버가 기록 시각으로 채움). 시각을
  지어내지 말 것.

## 3. 이미 기록한 것 확인
`list_recent_activity`(workspaceId)를 호출해 이 워크스페이스에 이미 남긴 활동을 본다.
각 항목의 `idempotencyKey`·`occurredAt`·`title`로 어떤 단위가 이미 있는지 판단한다.
**직전 기록 이후 새로 한 작업만** 남기면 된다 — 이미 있는 단위를 통째로 다시 만들지 말 것.

## 4. 단위로 나누고 키 정하기
분할 단위는 **의미 있는 작업 덩어리(하위 주제·마일스톤)**다. 단위마다 `log_activity`를
한 번씩 호출하고, 각 건의 `occurredAt`은 그 단위가 시작된 시각(2번)으로 잡는다.

- **뚜렷이 다른 주제 여러 개**를 다뤘으면 주제마다 나눈다.
- 주제가 하나여도 **분량이 크고 독립적으로 제목 붙일 만한 산출물이 여럿**이면 그 단위로
  나눈다. 분량/길이는 "쪼갤지 검토하라"는 신호일 뿐, 경계는 **의미 단위**로 잡는다.
- 같은 작업을 잠깐 떠났다 다시 이어가는 정도면 **한 건으로 합치고** 처음 시각을 쓴다.
- 글자 수만으로 기계적으로 쪼개지 말 것 — 각 조각은 그 자체로 의미 있는 결과를 가져야 한다.
- 단일 단위면 그냥 한 건으로 남긴다.

**키 규약** — `idempotencyKey`는 각 단위의 `occurredAt`에 고정한다(예:
`<세션>-2026-06-18T03:00:00Z`). 그러면 같은 단위는 항상 같은 키가 된다:
- 3번에서 **이미 있던 단위**면 그 `idempotencyKey`를 그대로 써서 **갱신**한다(내용이 바뀌었으면
  최신 요약으로 덮어쓴다).
- **새 단위**면 새 키로 남긴다.
- `<세션>` 부분은 이 대화 transcript 파일명(세션 id) 등 재실행해도 변하지 않는 값을 쓴다.

## 5. 요약 규칙 (건마다)
- **title**: 60자 이내 한국어, 명사형 종결('~ 추가', '~ 변경', '~ 개선').
  '~했다' 같은 서술형 어미와 함수명·파일경로 등 코드 식별자 금지.
- **summary**: 1~2개 구절, 개조식 명사형 종결. 마크다운·백틱 금지.
- **points**: 변경이 여러 갈래일 때만 2~5개의 개조식 명사형 구절. 단일이면 생략.
- **detail**: 보고서용 상세 기록(마크다운 허용, 코드 식별자 사용 가능).
- 원문 대화 transcript는 보내지 말고 요약만 보낸다.
